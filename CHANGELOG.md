# Changelog

All notable changes to The Marvellous Suspender are tracked in this file.

Format inspired by [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
The project adheres (pragmatically) to [Semantic Versioning](https://semver.org/).

Entries under "Unreleased" live on a feature branch until merged into `master`.

## [Unreleased] — feature/session-backup

### Added
- **Persistent log buffer and diagnostic report** (`debug.html`, `debug.js`, `gsUtils.js`): errors are now always captured to an in-memory ring buffer (max 500 entries) and flushed to `chrome.storage.local`, surviving Service Worker restarts. Warnings and verbose `log()` calls are captured when the new **captureLogs** flag is enabled. The flag persists to storage so it survives SW restarts; toggling it from the Diagnostic page immediately wakes the SW via `chrome.runtime.sendMessage`. `debug.html` is now a proper **Diagnostic page** with a log viewer (colour-coded by level), "Refresh", "Clear log", "Copy report" and "Download report" buttons. The report bundles extension version, browser user-agent, current tab profiler snapshot and the full log buffer into a single shareable text file — making it actionable for end-users who cannot access DevTools.
- `gsStorage.CAPTURE_LOGS` (`gsCaptureVerbose`) and `gsStorage.LOG_BUFFER` (`gsLogBuffer`) storage key constants.

### Changed
- **`debug.html` → Diagnostic page**: removed the defunct `debugInfo`/`debugError` DevTools-only toggles (they only affected the page's own JS instance, not the Service Worker). Replaced with the `captureLogs` persistent toggle. Instructions no longer mention "background page" (MV2 terminology). Sections restructured: Controls → Log buffer → Tab profiler.
- **`gsUtils.error()`**: always buffers to storage regardless of `debugError` flag; error path no longer has a dead commented-out else branch.
- **`gsUtils.warning()`**: stack-trace append inlined into `console.warn` call (no more mutation of `args` array).
- **`background.js`**: reads `gsCaptureVerbose` from storage at Service Worker startup to restore the flag; adds `setCaptureLogs` message handler. Removed large commented-out developer-only debug block.
- **Locale** (`en`, `it`): updated `html_about_debugging_prefix` / `html_about_debugging_suffix` to describe the diagnostic report workflow instead of the old "set debugging flags" copy.

### Fixed
- **`gsTabCheckManager.checkSuspendedTab()` — grouped suspended tabs lost after Chrome restart**: Chrome's tab-group replacement bug (`crbug.com/522338670`) can navigate a suspended tab's reload to `chrome://newtab/` instead of restoring `suspended.html`. When the initialisation check detects no active extension context after a resuspend attempt, it now fetches the tab's current state: if the reload navigated away from the suspended URL and the tab was in a group, a fresh tab is created with the original suspended URL, added to the same group, and the broken tab is removed. Pressing Back in the browser to recover the suspended page is no longer necessary.
- **`gsUtils.isSpecialTab()` — extension pages treated as normal tabs**: `chrome-extension://` URLs were not matched by `isBrowserInternalURL` (whose regex tests for `chrome:`, not `chrome-extension:`), so TMS's own pages (`debug.html`, `about.html`, `options.html`, …) were classified as normal tabs. This caused the check queue to repeatedly attempt content-script injection (failing with "Cannot access contents of url"), assign a suspension timer, and leave them as `unknown` in the profiler. A dedicated `startsWith('chrome-extension://')` guard is now applied before the `isBrowserInternalURL` call.
- **`tgs.getDebugInfo()` — `unknown` status for whitelisted/pinned/audible tabs**: when the content script did not respond (tab loading, whitelisted, pinned, or other), the profiler was short-circuiting and leaving the status as `unknown` instead of falling through to `calculateTabStatus`. Now `calculateTabStatus` is always called, passing the content-script status only when available, so the profiler correctly shows `whitelisted`, `pinned`, `audible`, `active`, `never`, etc.
- **`tgs.getDebugInfo()`**: restored the suppressed `gsUtils.warning()` call on content-script request failure.

### Added
- **"Wake whitelisted tabs" action**: new command that finds all suspended tabs whose original URL matches the whitelist and unsuspends them. Available in the extension popup (below "Unsuspend all tabs") and in the Options page next to the "Test whitelist" link (separated by a `·`). Implemented in `tgs.unsuspendWhitelistedTabs()`, routed via `background.js` on action `unsuspendWhitelisted`.
- `en` and `it` locale files: added `html_options_whitelist_unsuspend` and `html_popup_unsuspend_whitelisted_tabs`.
- **Automatic session backup** (`src/js/gsBackup.js`): new module that silently saves the current session as a JSON file to the user's default Downloads folder (inside a `tms-backups/` subfolder) at a configurable interval. Uses `chrome.downloads` API with `saveAs: false` and `conflictAction: 'overwrite'` so no dialog appears. Compatible with MV3 service workers (uses a `data:` URL instead of a Blob URL).
- `manifest.json`: added `"downloads"` permission, required to write files to disk from the background service worker.
- `gsStorage.AUTO_BACKUP_ENABLED` / `gsStorage.AUTO_BACKUP_INTERVAL`: two new settings keys (defaults: `false` / `'1'` hour).
- **Backup settings UI** in `options.html`: new "Automatic session backup" section (id `#section-backup`) with: enable checkbox, plain-text description below it (`.sectionDesc`, no box), interval selector (15min, 30min, 1h, 2h, 4h, 8h, 24h) and "Test backup now" button on the same row (`.backupIntervalRow`). The test button detects whether Chrome will download silently: it checks the download state after 1.5 s and shows either a reassurance message or a warning to disable "Ask where to save" in Chrome settings. Backup filenames include date and time (`tms-session-YYYY-MM-DDTHH-MM.json`). Up to 10 backups are retained; older ones are deleted automatically via `chrome.downloads.removeFile()` + `erase()`.
- **In-page section nav** in `options.html` and `history.html`: sticky tab-underline nav that stays pinned at the top of the content area while scrolling. The active tab updates automatically via a scroll listener (highlights the section currently in view). Nav styles live in `style.css` (shared). History sections received stable `id` attributes (`section-current/recent/saved/migrate`).
- **Back-to-top button** in `options.html`: fixed button in the bottom-right corner, fades in after 200 px of scroll and scrolls back to top smoothly on click.
- **Backup settings link** in `history.html`: small note at the top of the Sessions page pointing to `options.html#section-backup` for discoverability.
- `en` and `it` locale files: added `html_options_backup_title/enable/enable_tooltip_line1+2/interval/note/now`, `html_options_back_to_top`, `html_history_backup_settings_link`, `js_options_backup_now_running/done/error`, `js_options_backup_next_run`, `js_options_backup_file_count`.
- **Backup metadata** in `options.html`: below the interval row, a small muted line shows the next scheduled run time (from `chrome.alarms`) and the number of backup files currently archived on disk (from `chrome.downloads`). Both update live when the interval or enable state changes, and after a manual test backup.
- **Google Drive backup destination** (`src/js/gsBackup.js`): users can now choose between saving backups locally (Downloads folder) or to Google Drive. When Drive is selected, the extension authenticates silently via `chrome.identity.getAuthToken()` (OAuth 2.0, `drive.file` scope) and uploads the session JSON to a visible "TMS Backups" folder in the user's Drive via the Drive REST API. Cleanup of old Drive backups (keeping the 10 most recent) is handled automatically. The local `chrome.downloads` path is unchanged for users who prefer it.
- `manifest.json`: added `"identity"` permission and `"oauth2"` block (client_id placeholder, `drive.file` scope); updated CSP `connect-src` to allow `https://www.googleapis.com` and `https://accounts.google.com`.
- `gsStorage.AUTO_BACKUP_DESTINATION`: new setting key (default: `'local'`; values: `'local'` | `'drive'`).
- **Drive destination UI** in `options.html`/`options.js`: radio buttons to choose Local vs Google Drive; "Connect Google Account" / "Disconnect" buttons; live connected-account email display. The Drive auth panel shows/hides automatically based on the selected destination.
- `en` and `it` locale files: added `html_options_backup_destination`, `html_options_backup_dest_local/drive`, `html_options_backup_drive_connect/disconnect`, `html_options_backup_drive_account`, `js_options_backup_drive_connecting/auth_error`; updated `html_options_backup_enable`, `html_options_backup_enable_tooltip_line1+2` and `html_options_backup_note` to be destination-agnostic.
- **Settings export / import** (`options.html` / `options.js`): two buttons in the General section allow exporting all extension settings to a dated JSON file (`tms-settings-YYYY-MM-DD.json`) and importing them back. Import validates the file, asks for confirmation, merges only known keys against defaults, saves and syncs, then reloads the page. Prevents data loss during reinstalls or developer reloads.

- **Manual backup button in popup**: when automatic session backup is enabled (local or Drive), a new entry appears in the popup above the Settings/Session manager links. The label includes the active destination: "Backup session now (Local)" or "Backup session now (Cloud)". Clicking it fires an immediate `performBackup()` via the `backupNow` message action and closes the popup. Hidden when backup is disabled. New locale keys: `html_popup_backup_now_local`, `html_popup_backup_now_cloud`.
