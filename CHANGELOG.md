# Changelog

All notable changes to The Marvellous Suspender are tracked in this file.

Format inspired by [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
The project adheres (pragmatically) to [Semantic Versioning](https://semver.org/).

Entries under "Unreleased" live on a feature branch until merged into `master`.

## [Unreleased] — feature/session-backup

### Added
- **News feed** (`news.html`, `news.js`, `gsNewsFeed.js`, `gsNavBadge.js`): new dedicated "News" page in the sidebar that fetches and displays the latest TMS articles from the Marvellous Codeworks blog (`kb.marvellouscode.works/blog/rss.xml`). Each card shows title, excerpt, publication date, and a link to the full article. Details:
  - Feed cached in `chrome.storage.local` (TTL 6 h); periodic refresh via `chrome.alarms` in the service worker (`tms-news-feed`, every 6 h). Opening `news.html` also triggers an immediate `fetchAndCacheIfStale()` directly from the page.
  - Only articles tagged `"The Marvellous Suspender"` via RSS `<category>` are shown — articles from other Marvellous Codeworks products (e.g. TGD) are filtered out.
  - Maximum 10 items displayed, laid out in a 2-column grid.
  - A red badge dot on the "News" sidebar entry signals unread articles; opening the page marks all items as seen. Badge logic lives in the shared `gsNavBadge.js` module, loaded by every extension page.
  - `manifest.json`: added `https://kb.marvellouscode.works` to `connect-src` (required for both extension pages and the service worker under MV3).
- New i18n keys (`html_sidebar_news`, `html_news_title`, `html_news_desc`, `html_news_loading`, `html_news_empty`) in `en` and `it` locales.

- **News feed: per-article read tracking** (`news.js`, `news.html`, `gsNewsFeed.js`, `style.css`): articles are no longer marked as read simply by opening the News page. Unread articles are highlighted with a blue left border and a "NEW" pill in the date area. Clicking the article title link marks that single article as read and removes the highlight. A "Mark all as read" link appears top-right above the grid when unread items exist; it disappears once everything is read. `gsNewsFeed.markSeen(link)` added as a new public function for single-item tracking. New i18n keys `html_news_new_pill` and `html_news_mark_all_read` in `en` and `it`.

- **News feed alarm: per-install jitter within working hours** (`gsNewsFeed.js`): reduced to once per day (`periodInMinutes: 1440`, `CACHE_TTL_MS: 24h`). The fetch time is random but fixed per installation — generated once within the 08:00–20:00 local window and persisted in `tmsNewsFeedMinuteOffset`. Spreads ~100k users across 720 minutes (~139 req/min), avoiding coordinated bursts on the GitHub Pages CDN. The debug page shows the assigned daily time (e.g. "daily at 14:23 local").

- **Debug: news feed card** (`debug.html`, `debug.js`): new card in the Controls section of the Debugger page. Shows last fetch timestamp, unread/total article count, next scheduled alarm time, and per-install jitter slot (e.g. "daily at 14:23 local"). The "force refresh" link bypasses the TTL and triggers an immediate RSS fetch; it is hidden on Chrome Web Store installs (`update_url` check) and visible only on unpacked developer builds. Action links ("run claim", "force refresh") are positioned top-right in the card header, matching the style of the other debug controls.
- **RSS icon** (`img/icons.svg`): added `#rss` symbol (Lucide, three arcs + dot) to the shared sprite so it can be referenced as `<use href="img/icons.svg#rss"/>` from any extension page.

- **Donate bar in sidebar** (`gsDonateLinks.js`, `style.css`): a compact row of 3 favicon-sized icons (GitHub Sponsors, Ko-fi, Buy Me a Coffee) now appears at the bottom of the left navigation sidebar on every extension page. Icons are bundled locally (`img/favicon-github.png`, `img/favicon-kofi.ico`, `img/favicon-buymeacoffee.ico`) to comply with the extension's `img-src 'self'` CSP. Rendered at 16 px with 45 % default opacity, full opacity on hover. Accessible via `title` and `aria-label` attributes.

- **Support development section** (`about.html`): added a "Support development" section to the About page, placed between the issue reporting/debugging links and the Credits block. Lists three donation platforms — GitHub Sponsors, Ko-fi, and Buy Me a Coffee — with a brief introductory sentence. New i18n keys `html_about_section_support` and `html_about_support_text` added to `en` and `it` locales.

- **Restore from backup** (`backup.html`, `backup.js`, `gsBackup.js`): new "Restore from backup" section at the bottom of the Backup page. Users can now recover a session directly from this page without manually navigating to the Sessions tab.
  - **From a local file**: a file picker loads any `tms-session-*.json` backup file, parses it, and imports it into IndexedDB as a saved session (auto-named from the filename, e.g. "Backup 2026-06-24 09:30"). A link to `history.html` appears inline so the user can immediately open Sessions to restore the desired windows.
  - **From Google Drive** (shown only when Drive is connected): "Load Drive backups" fetches the list of backup files from the `TMS Backups` Drive folder, populates a dropdown, and lets the user import the selected file with a single click. Same import-and-link flow as the local path.
- `gsBackup.listDriveBackups()`, `gsBackup.downloadDriveBackupContent(fileId)`, `gsBackup.importBackupJson(jsonText, sourceName)` — three new public functions to support the restore flow.
- New i18n keys (`html_backup_restore_*`, `js_backup_restore_*`) in `en` and `it` locales.
- `gsUtils.localiseHtml()`: support for `data-i18n-aria-label` attribute — sets `aria-label` from the i18n message key, enabling accessible labels on form controls without hardcoded strings.

- **Multi-device Drive backup: per-device rotation and friendly names** (`gsBackup.js`, `gsStorage.js`, `backup.html`, `backup.js`, `backup.css`): Drive backups are now device-aware. Each Chrome installation generates a stable 8-char hex `deviceId` stored in `chrome.storage.local`. Backup filenames embed the deviceId (e.g. `tms-session-a3f7b2c8-2026-06-29T10-30.json`) so a session saved from a lightweight secondary device (e.g. MacBook with few tabs) can never evict backups from the primary workstation. Rotation now applies per-device: each device keeps its own independent pool of up to N recent backups. Legacy backups (files created before this change, with no deviceId in the filename) are preserved and excluded from rotation. Key details:
  - **Per-device rotation (Drive only)**: `cleanupOldDriveBackups` groups files by `deviceId` prefix and prunes each group independently. Local backups are naturally per-machine (each device has its own Downloads folder), so `cleanupOldLocalBackups` uses simple N-most-recent pruning with the original date-only filename format — no deviceId needed.
  - **Device registry**: on every Drive backup, `tms-device-{deviceId}.json` is created/updated in `appDataFolder` with `{ name, lastSeen }`. This provides a lightweight record of which devices have ever backed up to this Drive account.
  - **Configurable limit**: a new "Max backups to keep" number input (min 2, max 50, default 10) on the Backup page controls retention for both destinations. For Drive the limit applies per device; for local it applies to the total on that machine. Stored in `chrome.storage.sync` via the new `AUTO_BACKUP_MAX_FILES` key.
  - **Friendly device name** (multi-device only): when the Backup page detects that more than one device has backed up to the same Drive account, a "Device name" text field appears so the user can label the current machine (e.g. "MacBook", "PC lavoro"). The name is saved in `chrome.storage.local` and written to the device registry on the next backup. The field is hidden on single-device setups to avoid unnecessary UI clutter.
  - **Restore UI**: the Drive restore dropdown groups backups by device (using `<optgroup>` with the device friendly name as label) when multiple devices are detected; legacy backups appear in a separate "Legacy backups" group. Single-device users see a flat list unchanged from before.
  - New i18n keys (`html_backup_max_files_label`, `html_backup_device_name_label`, `html_backup_device_name_placeholder`, `html_backup_device_name_save`, `js_backup_restore_legacy_group`) in `en` and `it` locales.
  - `html_options_backup_note` updated to reflect the new filename format.

- **Append original URL to suspended tab title** (`gsStorage.js`, `options.js`, `options.html`, `suspended.js`): new option (enabled by default) that appends the original URL to `document.title` of suspended pages so Chrome's "Search Tabs" (Ctrl+Shift+A) can find them by hostname. The visual title displayed inside the suspended page is unchanged. Option is exposed as a checkbox in the "Suspended tabs" section of the Options page. Closes [#249](https://github.com/gioxx/MarvellousSuspender/issues/249).
- New i18n keys (`html_options_suspended_append_url_to_title`, `…_tooltip_line1`, `…_tooltip_line2`) in `en` and `it` locales.

- **Drive backup files: download panel** (`backup.html`, `backup.js`, `gsBackup.js`, `backup.css`): collapsible `<details>` panel inside the "Restore from backup" section, shown only when Drive is connected and at least one backup file exists. Since Drive backups are stored in the hidden `appDataFolder`, users have no way to access them via the standard Drive UI; this panel fills the gap. The file list (name, date, size) is populated by `updateDriveAuthUI` reusing data already fetched — no extra API call on open. Each row has a "Download" button; files are grouped by device name when multiple devices are present, matching the existing restore dropdown behaviour. A download triggers `gsBackup.downloadDriveBackupAsFile(fileId, filename)`, which fetches the JSON content from Drive and saves it via `chrome.downloads.download` into the `tms-backups/` subfolder.
  - `gsBackup.downloadDriveBackupAsFile(fileId, filename)` — new public function.
  - `gsBackup.listDriveBackups()` — now also requests the `size` field from the Drive API.
  - New i18n keys (`html_backup_drive_files_title`, `html_backup_drive_download_btn`, `html_backup_drive_no_files`, `html_backup_drive_files_loading`) in `en` and `it` locales.
- **Debug: backup device identity card** (`debug.html`, `debug.js`, `debug.css`): new card in the Controls section showing the current device's backup ID (`gsBackupDeviceId`, 8-char hex) and friendly name. The ID is rendered in monospace with `user-select: all` so it can be copied in one click for support purposes. Populated at page load via `gsBackup.getDeviceId()` and `gsBackup.getDeviceName()`.
- **Per-element "Saved" feedback in Options** (`options.js`, `style.css`): every option on the Options page now shows a brief "Salvato / Saved" confirmation inline after the changed control when the value is actually stored. The feedback fades in and disappears after 2 seconds; language changes are excluded (page reloads immediately). Implemented via dynamic span injection — no HTML changes required.

### Added
- **Settings backup/restore cards** (`backup.html`, `backup.js`, `gsBackup.js`): the flat "Settings backup" section is redesigned with two cards matching the session-restore layout. The **Local card** always shows Export and Import buttons. The **Google Drive card** appears only when Drive is connected and offers both "Save settings to Drive" (existing) and the new **"Restore settings from Drive"** button, which downloads `tms-settings.json` from the `TMS Backups` Drive folder, validates it, and applies it after user confirmation. The settings section is moved to the bottom of the page so session backup and restore have primary prominence.
- `gsBackup.getDriveSettingsInfo()` — lightweight metadata query that returns `{ id, modifiedTime }` for `tms-settings.json` on Drive, or `null` if absent. Used to display the last-backup date in the Drive settings card.
- `gsBackup.downloadDriveSettingsContent()` — fetches `tms-settings.json` from Drive; throws `TMS_SETTINGS_NOT_FOUND` if the file does not exist. Internally shares `_findDriveSettingsFile()` with `getDriveSettingsInfo()` to avoid duplicate folder queries.
- `applySettingsJson(jsonText)` extracted as a shared internal helper in `backup.js` — used by both the local file import and the new Drive restore handler, eliminating duplication.
- **Overwrite protection for Drive settings backup** (`gsBackup.js`, `backup.js`): saving settings to Drive now has two safety layers. **Option A** (UI): before overwriting an existing `tms-settings.json`, the "Save settings to Drive" button shows a confirmation dialog with the date of the existing backup ("Existing settings backup from DD/MM/YYYY HH:MM. Overwrite?"). If the user cancels, no action is taken. **Option B** (silent): `performDriveSettingsBackup()` copies the current `tms-settings.json` to `tms-settings-prev.json` before overwriting, so the previous version is recoverable from Drive even after a mistaken save. Private helper `_writeDriveFile(token, folderId, filename, jsonString)` extracted to support both files without code duplication. After a successful save the Drive-card date badge updates immediately.
- New i18n keys (`html_backup_settings_local_title`, `html_backup_settings_drive_title`, `html_backup_settings_drive_restore`, `js_backup_settings_drive_downloading`, `js_backup_settings_drive_not_found`, `js_backup_settings_drive_restore_success`, `js_backup_settings_drive_restore_error`, `js_backup_settings_drive_overwrite_confirm`) in `en` and `it` locales.

### Changed

## [9.0.0] — 2026-06-24

### Changed
- **`updated.html` / `updated.js`**: after the automatic tab-restore completes, `suspendy-guy.webp` (in-progress) is now swapped to `suspendy-guy-success.webp` (success) via `toggleUpdated()`.
- **`gsUtils.js`**: added `getMessage(key)` helper that reads from `_localeMessages` first (custom language override) and falls back to `chrome.i18n.getMessage()`, ensuring the user-chosen language is respected everywhere.
- **`popup.js`**: replaced all `chrome.i18n.getMessage()` calls with `gsUtils.getMessage()` so that the status detail text and "Backup now" label honour the language selected in options rather than the browser UI locale.

### Added
- **Persistent log buffer and diagnostic report** (`debug.html`, `debug.js`, `gsUtils.js`): errors are now always captured to an in-memory ring buffer (max 500 entries) and flushed to `chrome.storage.local`, surviving Service Worker restarts. Warnings and verbose `log()` calls are captured when the new **captureLogs** flag is enabled. The flag persists to storage so it survives SW restarts; toggling it from the Diagnostic page immediately wakes the SW via `chrome.runtime.sendMessage`. `debug.html` is now a proper **Diagnostic page** with a log viewer (colour-coded by level), "Refresh", "Clear log", "Copy report" and "Download report" buttons. The report bundles extension version, browser user-agent, current tab profiler snapshot and the full log buffer into a single shareable text file — making it actionable for end-users who cannot access DevTools.
- `gsStorage.CAPTURE_LOGS` (`gsCaptureVerbose`) and `gsStorage.LOG_BUFFER` (`gsLogBuffer`) storage key constants.

### Changed
- **`debug.html` → Diagnostic page**: removed the defunct `debugInfo`/`debugError` DevTools-only toggles (they only affected the page's own JS instance, not the Service Worker). Replaced with the `captureLogs` persistent toggle. Instructions no longer mention "background page" (MV2 terminology). Sections restructured: Controls → Log buffer → Tab profiler.
- **`gsUtils.error()`**: always buffers to storage regardless of `debugError` flag; error path no longer has a dead commented-out else branch.
- **`gsUtils.warning()`**: stack-trace append inlined into `console.warn` call (no more mutation of `args` array).
- **`background.js`**: reads `gsCaptureVerbose` from storage at Service Worker startup to restore the flag; adds `setCaptureLogs` message handler. Removed large commented-out developer-only debug block.
- **Locale** (`en`, `it`): updated `html_about_debugging_prefix` / `html_about_debugging_suffix` to describe the diagnostic report workflow instead of the old "set debugging flags" copy.

### Fixed
- **`gsTabCheckManager.checkSuspendedTab()` — grouped suspended tabs lost after Chrome restart**: Chrome's tab-group replacement bug (`crbug.com/522338670`) can navigate a suspended tab's reload to `chrome://newtab/` instead of restoring `suspended.html`. When the initialisation check detects no active extension context after a resuspend attempt, it now fetches the tab's current state: if the reload navigated away from the suspended URL and the tab was in a group, a fresh tab is created with the original suspended URL, added to the same group, and the broken tab is removed. Pressing Back in the browser to recover the suspended page is no longer necessary.
- **`gsUtils.isSpecialTab()` — extension pages treated as normal tabs**: `chrome-extension://` URLs were not matched by `isBrowserInternalURL` (whose regex tests for `chrome:`, not `chrome-extension:`), so TMS's own pages (`debug.html`, `about.html`, `options.html`, …) were classified as normal tabs. This caused the check queue to repeatedly attempt content-script injection (failing with "Cannot access contents of url"), assign a suspension timer, and leave them as `unknown` in the profiler. A dedicated `startsWith('chrome-extension://')` guard is now applied before the `isBrowserInternalURL` call.
- **`tgs.getDebugInfo()` — `unknown` status for whitelisted/pinned/audible tabs**: when the content script did not respond (tab loading, whitelisted, pinned, or other), the profiler was short-circuiting and leaving the status as `unknown` instead of falling through to `calculateTabStatus`. Now `calculateTabStatus` is always called, passing the content-script status only when available, so the profiler correctly shows `whitelisted`, `pinned`, `audible`, `active`, `never`, etc.
- **`tgs.getDebugInfo()`**: restored the suppressed `gsUtils.warning()` call on content-script request failure.

### Removed
- Dead `debugWindowId` `innerHTML` assignments (4 commented-out lines) from `historyUtils.js`.
